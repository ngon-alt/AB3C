戦略指南 AI 楽天版 ― v1 仕様書

作成日: 2026-06-12 位置づけ: 実装着手前の確認用仕様書。権さんのレビュー・承認を経てから実装に入る。 前提資料: docs/楽天市場-商品分析機能-構想ドキュメント.md（2026-05-31 の設計議論）、読書ノート projects\docs\notes\sem_senryaku_note.md・4model_note.md スコープ: 約1週間でリリースする v1。v2 以降に送る項目は第9章に明記。

1. 目的とコンセプト

楽天市場の店長が、自分の商品LPのURLを入力すると、次の三段が一本の物語として返ってくる。

段 問い 中身
第一段 売れている店は、なぜ売れているのか 競合上位商品のレビューから「選ばれる理由」を採掘して見せる
第二段 自分は、なぜ売れないのか 自社LPの訴求と顧客評価のズレ、競合との強み弱み比較、LPの語り方の欠落
第三段 では、どうすればいいのか 訴求ポイントの再設定 → 戦略メッセージ確定 → コンテンツ増強の処方箋

分析部品を並べるレポートではなく、「なぜ → なぜ → ならば」の物語として書く。これは戦略指南AI本体で起きた「経営・戦略の知識がない人には中身が理解できない」問題への直接の答えである。店長は自分の商売の物語として読める。

設計の根本前提（権さん確認済み・2026-06-12）

  • ニーズ/ウォンツは商品の属性ではなく、購入者の段階である。 同じミネラルウォーターでも「ミネラルウォーター」と検索する人（ニーズ段階）と「ボルヴィック」と検索する人（ウォンツ段階）がいる。
  • ブランド品であればウォンツ段階の購入者が多いだろうと言えるが、楽天で売られる商品の大半はニーズ段階の購入者が多い。
  • 「ニーズ型前提で統一する」の真意は、商品の分類ではなく「説得の設計」の統一である（権さん補足・2026-06-12）:
  • ウォンツ型（型番・指名買い）の商品は、商品本体は競合とスペックが同一のため、価格・納期だけで判断される。
  • しかし仕入れ品の売り手でも、サービス（購入前・購入中・購入後）と「外側のパッケージ」は自分たちの提供物であり、ここで差別化が作れる。例: 大型家具の購入時、有害物質付着で通常廃棄できない旧家具を回収・廃棄してくれる店——サービスによるAdvantageの実例。外側のパッケージの例: おまけの同梱、配送用段ボールの工夫、剥がしやすいガムテープの貼り方。
  • 用語注意（権さん指摘・2026-06-12）: ここで言う「外側のパッケージ」は、価値サークルの「パッケージング」軸とは別物。価値サークルのパッケージングは商品そのもののベースパッケージ（商品の一部）を指す。仕入れ品の売り手が工夫できるのは、その外側にある包装・同梱・配送まわり＝サービスに近い領域である。
  • そしてサービスの価値は黙っていては伝わらない。コンテンツで説明して選んでもらう＝ニーズ型と同じ「コンテンツによる説得」の設計が必要になる。同じ商品を売る店の中から「ここのサービスがいい」と選んでもらう構造。
  • だから全商品を同じエンジン（ニーズ型の説得設計）で分析する。入口での機械的なモデル判定・処理分岐は置かない。ウォンツ型商品は商品本体のスペック差が抽出されないため、分析結果は自然にサービス・パッケージング軸の強みに収束する（構想ドキュメント第3章の合意どおり）。
  • ただし説明の言葉として「ウォンツ型」を使うのは推奨。型番・指名買い色の強い商品には「この商品はウォンツ型（指名買い）のため、商品本体は価格・納期で判断されます。差別化はサービスで作り、その価値をコンテンツで伝えて選んでもらいましょう」という言い方をする。分類用語としてではなく、店長への説明をわかりやすくする道具として使う。
  • ニーズ段階前提の理論的裏付け＝「モール出店は、他店の同種商品とおのずと比較され、相対的なウォンツが形成される装置」（『4モデル式』p.88）。楽天の商品LPは構造的に比較シーンの中で読まれるページであり、分析は単独ページの良し悪しではなく競合との相対評価で行う。

2. 使い手

楽天ショップの店長本人。 コンサル・代理店ではなく、店長が自分で画面を見ながら企画を考える道具として設計する（構想ドキュメント第7章）。文字サイズ・日本語ラベル等は戦略指南AI本体のルール（本文18px以上・英語ステータス語禁止）をそのまま適用する。

3. 入力仕様

3-1. 入力の流れ

① 自社の商品LP URL を入力 ② カテゴリーを指定（前回値が自動で入る。上書き可） ③ 競合候補リストが提示される → チェックで選ぶ（外す・足す・自由URL追加可） ④ 分析実行

3-2. カテゴリー登録（大・中・小の3階層）

  • 大カテゴリー・中カテゴリー・小カテゴリーの3階層を選択させる（権さん指定・2026-06-12）。
  • 店長本人に指定させる。操作の手間ではなく「自分の商品はどの戦場にいるのか」を意識させる教育効果を狙う（構想ドキュメント第7章と同思想）。3階層を自分で選ぶこと自体が「大カテゴリーでは激戦のレッドオーシャン、小カテゴリーなら競合数件のニッチ」という戦場の構造を見る体験になる。
  • 店舗単位でデフォルト保持: 一度入力したカテゴリー（3階層とも）はサイト（店舗）に紐づけて保存し、次回の商品URL入力時に自動で埋める。消したいときは上書きできる。小規模店はほぼ全商品が同一カテゴリーという実態に合わせる。
  • カテゴリーは競合母集団の特定（3-3）と、価値軸の第2層（4-2）の両方に使う。競合候補の提示はまず小カテゴリーのランキングを基本とし、必要に応じて中・大に広げる（粒度の使い分けの詳細は実装時に調整）。

3-3. 競合候補の自動提示 → 店長が主体的に選ぶ

  • 指定カテゴリーのランキングから競合候補を自動で並べて見せ、店長がチェックボックスで競合を確定する。
  • 候補には順位・価格・レビュー件数・平均星数（★）を添える。「1位だが価格が1/3（比較対象として不適かも）」が一目で分かるようにする。ランキング上位＝適切な競合とは限らない（激安上位は学べない）ため、比較に値するかの判断は店長に残す。
  • 平均星数は選択基準というより学習のため（権さん指定・2026-06-12）。件数×星数の組み合わせから市場の構造が見える（例: 件数多×★3.8＝売れているが不満も多い／件数少×★4.8＝小さいが熱烈に支持される）。レビュー分析に入る前の「この競合はどんな選ばれ方をしているか」の予告編として機能する。公式APIがreviewCount・reviewAverageを両方返すため追加取得コストはない。
  • 自由URL追加を許す。「本当はこれと比べたい」という指名比較に対応する。
  • この「候補から選ぶ」行為自体が、自分の戦場を眺める教育体験になる。
  • 取得経路はランキングの公式API（楽天ウェブサービス）を本命とする（第7章）。
  • 競合数の目安: 自社1 + 競合3〜5商品（レビュー取得・トークンコストとのバランス。上限は実装時に調整）。

4. 分析パイプライン

[自社LP取得] ──┐ [競合LP取得] ──┼→ [レビュー層化抽出] → [価値軸の仕分け] → [ズレ可視化] → [AB3C・戦略メッセージ] → [三段の物語レポート] [レビュー取得] ─┘

4-1. レビューの層化抽出（分析の心臓）

  • 各商品につき、件数を均等に取るのではなく評価で層化する: 高評価（★4〜5）から数十件＋低評価（★1〜2）から数十件。「選ばれる理由」と「離れる理由」のコントラストが少ない件数で鮮明に出る（構想ドキュメント第8章）。
  • 商品ごとに一度要約してから束ねる二段構え（コンテキスト上限対策）。
  • トークンコストは1分析あたり数百円のオーダーで制約にならない。件数を絞る理由はコストではなく精度と鮮度。

4-2. 価値軸の抽出と仕分け（価値サークル6軸）

レビューの一文一文を権さんの価値サークルの6軸に仕分けて、自社と競合の強み弱みを構造化する。

  • 商品力の3軸: 機能（商品本体）／デザイン／パッケージング
  • サービスの3軸: 購入前サービス／購入中サービス／購入後サービス

仕分け時の用語注意: 「パッケージング」軸は商品そのもののベースパッケージ（商品の一部。例: ギフト用化粧箱）を指す。配送用段ボール・梱包の丁寧さ・おまけ同梱・開封しやすさ等の「外側のパッケージ」への言及は、パッケージング軸ではなくサービス軸（主に購入中サービス）に仕分ける。仕入れ品の売り手が差別化できるのは後者（第1章の用語注意参照）。

価値軸（比較の横軸）は3層で構成する（構想ドキュメント第4章）: 1. 全通販共通項目（価格・納期）— 事前固定 2. カテゴリー固有項目（食べ物なら味、衣類ならサイズ感）— v1ではAIがカテゴリーから推定 3. その商品に固有の価値軸 — レビューから抽出

4-3. 訴求と評価のズレ可視化（v1の核）

  • 自社LPが訴求していること（LPテキストから抽出）と、顧客がレビューで実際に評価していること（4-1, 4-2の結果）を突き合わせる。
  • 二段階で見る: 1. 訴求と評価がずれていれば「訴求の力点が間違っていたのではないか」という気づき 2. その顧客評価が競合と比べて本当に強みか確認 — 競合もみな「おいしい」なら強みではない。自社だけ違うところに顧客が強みを認識しているなら、それは本物の差別的優位であり、伝えるだけで価値になる
  • レビューで繰り返される特徴語＝Advantage候補、という理論的裏付け: 購買率の高いキーワードは「商品の差別的特徴を表すキーワード」（『SEM戦略』p.81-82 の「基礎化粧品→ナノ化粧品」の論理の裏返し）。

4-4. LPの語り方診断【観点リストは再設計待ち・未確定】

ニーズ段階の購入者は「サイト訪問後に購買意欲を高められるか」が勝負であり、LPコンテンツの質が売上に直結する（『4モデル式』p.65）。そこで自社LPの「語り方」を観点リストで診断するパートを置く。

  • 出発点の素材は『4モデル式』p.145-146 のガイドコンテンツ6要素（①直観的にわかるベネフィット ②差別的優位点 ③商品詳細 ④使用方法・利用シーン ⑤ユーザーの声 ⑥信頼性）。
  • ただし権さんの判断（2026-06-12）: この6要素はそのままでは使わない。表現の見直しと要素の追加が必要。 楽天LPの実態（縦長画像主体・ファーストビュー・価格/送料/ポイント表示・ギフト対応等）に合わせた再設計を権さんと行ってから確定する。
  • v1実装スケジュールとの関係: 観点リストの確定が間に合わない場合、このパートはプロンプト内の差し替え可能な定義として実装し、リスト確定後に文言を入れ替える（コード構造はリストに依存させない）。

4-5. AB3C・戦略メッセージ（出口）

  • ズレ分析・強み弱みを踏まえ、選んだ軸をAdvantageに据えたAB3C（Benefit／Advantage／3C）と戦略メッセージを生成する。
  • 強み弱みは両方提示する。強い軸は訴求の中心に据えるよう背中を押し、弱い軸は「補うか、新しい強みを作るか」と問いを返す（Advantage不成立時は商品・サービス開発を提案する、というAB3C原則のとおり）。
  • 「商品を買う理由」と「この店で買う理由」を分けて言語化する（『4モデル式』p.16-18, 186）。楽天の同一商品多店舗環境では後者（サブベネフィット: 同梱・サポート・ギフト対応等）が効く。

5. 出力仕様（レポート構成）

三段の物語をそのまま章立てにする。

第一段「売れている店は、なぜ売れているのか」

  • 選んだ競合ごとに、レビューから採掘した「選ばれる理由」を提示（価値サークル6軸への仕分け結果つき）
  • 競合間で共通する強み（＝このカテゴリーの土俵）と、各競合に固有の強み（＝各社の山）を分けて見せる

第二段「自分は、なぜ売れないのか」

  • 自社の訴求 vs 顧客評価のズレ（4-3）
  • 競合との6軸比較表（強み弱み両出し）
  • LPの語り方診断（4-4・観点リスト確定後）

第三段「では、どうすればいいのか」

  • 訴求ポイントの再設定案（「訴求のバランスをユーザーの重視点に偏らせる」『4モデル式』p.80）
  • 戦略メッセージ（AB3C出口）
  • コンテンツ増強の優先順位
  • 「どの軸で戦うか」のシナリオ提示（複数の世界線。v1では簡易版＝主要2案程度）
  • 全社が低い／誰も載せていない軸があれば新しい軸の提案（ブルーオーシャン。近江牛.comのパッケージ事例の構造的再現）

表示・言語ルール

  • 戦略指南AI本体と同一: AB3Cカラー（C=黒 #1a1a14／B=赤 #FF0000／A=青 #1a6fd4）、本文18px以上、ok/warn/ng等の英語ステータス語をユーザーに見せない。
  • 評価語は「整っている」「まだ弱い」「改善が必要」等の自然な日本語。

6. 採用しない・参考にしないと決めたこと

項目 理由
タイトル（商品名）へのキーワード有無による競合の本気度判定 『SEM戦略』のサープス分析由来の手法だが、楽天内SEOとGoogle SEOは評価要素が大きく異なるため、タイトルのキーワードだけでは判断できない（権さん判断・2026-06-12）。本質的な競合を探すという視点のみ受け継ぎ、手法は移植しない
ニーズ/ウォンツの機械的モデル判定・処理分岐 第1章のとおり。全商品を「ニーズ型の説得設計」で統一分析（ウォンツ型は結果が自然にサービス軸へ収束）。「ウォンツ型」という言葉自体は店長への説明用語として使う
ランキング順位 vs 楽天サーチ検索順位の突合 楽天内SEOの評価要素に依存する診断のため、v1では扱わない
検索キーワードベースの競合発見 楽天はカテゴリー回遊が主流（構想ドキュメント第2章の結論どおり、競合発見はカテゴリーランキング軸）

7. 技術仕様

7-1. データ取得（検証結果: 2026-06-12 スパイク）

対象 経路 検証結果
レビュー本文 review.rakuten.co.jp/item/1/{shopId}_{itemId}/1.1/ を直接取得 成功。ブラウザ相当のUAヘッダー付きならHTTP 200。ページ内 window.__INITIAL_STATE__ のJSONにレビュー本文（body）・評価が構造化されて入っており、HTML解析不要
商品LP item.rakuten.co.jp を直接取得 未検証（実装初日に確認）
カテゴリーランキング 楽天ウェブサービス（公式API） ページ直接取得は403のためAPI経路が本命。アプリID取得が必要（無料）。APIのitemCodeからレビューページの数値ID（shopId_itemId）への橋渡しは実装初日に検証

リスクと留意点: - 検証はローカル回線から実施。VercelのデータセンターIPからは403になる可能性が残る。実装初日に本番経路で確認し、不可なら軽量プロキシ（中継）を挟む。 - 標準的なfetch（プログラム的UA）は403で弾かれるため、ブラウザ相当のUAヘッダーが必須。 - ページ販売単位のエンコーディングに注意（旧ページはEUC-JP混在の可能性）。

7-2. AI処理

  • モデル: claude-sonnet-4-6（本体と同じ）
  • レビューは商品ごとに要約→束ねる二段構え（4-1）
  • 分析プロンプトには読書ノート（sem_senryaku_note.md / 4model_note.md）から該当の方法論を組み込む
  • LP語り方診断の観点リストはプロンプト内の差し替え可能な定義として持つ（4-4）

7-3. 実装の位置づけ

  • AB3Cリポジトリ内の新機能として実装（別サービスとして切り出さない。v1は速度優先）
  • デプロイは通常フロー厳守: staging → preview.senryaku.ai で権さん確認 → main
  • updates.js への告知エントリ追加（リリース時）

9. 未決事項（権さんと詰める）

  1. LP語り方診断の観点リスト再設計（4-4）— ガイドコンテンツ6要素の表現見直し＋楽天LP実態に合わせた要素追加。権さんの検討待ち
  2. 点数化の客観性（戦略キャンバス図示の前提。v2課題）
  3. 価格・課金の扱い — 既存の戦略診断チケット/戦略指南サブスクとの関係、楽天版固有の価格を立てるか
  4. 公開誉め診断（プロモーション）との連携 — ジャンル単位比較コンテンツの生成に本機能をどう流用するか
  5. カテゴリー固有項目の作り方 — v1はAI推定で行くが、精度が出なければカテゴリー辞書方式を検討（構想ドキュメント第8章の論点1）
