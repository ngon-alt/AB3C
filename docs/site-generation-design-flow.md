# 戦略指南AI × ウェブサイト生成——設計思想と実装フロー

> 作成：2026-06-07（⑪セッション）  
> 用途：特許申請・機能設計・伴走支援者向け説明資料のベース

---

## 基本思想：ウェブサイト5階層概念モデル

本フローの設計基盤は、Jesse James Garrett が提唱した**ウェブサイト5階層概念モデル**（The Elements of User Experience, 2002）にある。

```
┌─────────────────────────────────┐  ← 表層（Surface）
│         視覚デザイン・UI         │    色・フォント・写真・レイアウト
├─────────────────────────────────┤
│      骨格（Skeleton）           │    ページ設計・ナビゲーション設計
├─────────────────────────────────┤
│      構造（Structure）          │    サイトマップ・情報アーキテクチャ
├─────────────────────────────────┤
│      要件（Scope）              │    コンテンツ要件・機能要件
├─────────────────────────────────┤  ← 基盤
│      戦略（Strategy）           │    事業目標・ユーザーニーズ
└─────────────────────────────────┘
```

**このモデルの本質は「下から積み上げる」ことにある。**  
表層（デザイン）から入ると、下層（戦略）との整合が取れなくなる。  
戦略から始めることで、サイト全体が一気通貫の論理構造を持つ。

### 5階層と本フローの対応

| 5階層 | 本フローのフェーズ | 戦略指南AIの役割 |
|---|---|---|
| **戦略** | Phase 1：AB3C分析・戦略メッセージ確定 | AIが戦略を言語化 |
| **要件** | Phase 2：コンテンツ企画 | 競合調査・プッシュ/プル・ユーザーモデル別設計 |
| **構造** | Phase 3：サイト構造設計 | ユーザーモデルからサイトマップを導出 |
| **骨格** | Phase 4：コンテンツ詳細化・担当決定 | 各ページのコンテンツ配置・工数見積り |
| **表層** | Phase 5：生成・実装 | デザインシステム確認→フルページ生成 |

**本フローが他のAIサイト生成と根本的に異なる点：**  
多くのAIサイト生成ツールは「表層から入る」——テンプレートを選んでコンテンツを流し込む。  
戦略指南AIは「戦略から入る」——AB3Cで言語化された戦略が、要件→構造→骨格→表層のすべてを規定する。  
だからこそ生成されたサイトに戦略が宿り、「どこにでもあるサイト」にならない。

---

## 思想の核心

```
浅いインプット × AI生成 = AI感のある一般論
深いAB3C × コンテンツ企画 × AI生成 = 戦略に根ざした固有のサイト
```

戦略指南AIの本質的差別化は「分析しっぱなしにしない」こと。  
AB3Cで戦略を言語化し、コンテンツ企画を経て、ウェブサイトという実装物まで**一気通貫**でつなぐ。

## Phase 1：戦略立案（戦略指南AIの既存機能）

### AB3C分析

戦略指南AIがURLまたはテキストから以下を抽出・言語化する。

| 要素 | 内容 |
|---|---|
| **Benefit** | 顧客が求める価値（ニーズ→ウォンツ） |
| **Advantage** | 差別的優位点——「競合より選ばれる理由」 |
| **Customer** | 顧客像・市場規模・購買段階（ニーズ/ウォンツ） |
| **Competitor** | 直接競合・間接競合 |
| **Company** | 強み・構造・情熱（Advantageの裏付け） |

### 戦略メッセージの確定

- BenefitとAdvantageを統合した核心メッセージ
- 「論理的な戦略言語」から「感情に届くキャッチコピー」への昇華（AI複数案→人間選択）

### ユーザーモデルの抽出

AB3CのCustomer分析から最大3パターンのユーザーモデルが生成される。

```
パターン1：最重要ユーザー（サイト全体の骨格を担う）
パターン2：近いユーザー（共通部分が多い）
パターン3：遠いユーザー（独立した動線が必要）
```

### ヒーロー設計のB/A優先度（自動判断）

競合度 × Customer段階（ニーズ/ウォンツ）から導出する。

| 競合状況 | Customer段階 | ヒーロー設計 |
|---|---|---|
| 競合少（新市場） | 問わず | **Benefit主体**——「あなたが求めるものはここにある」 |
| 競合多（成熟市場） | ウォンツ段階 | **Advantage主体**——「なぜ我々か」を先に答える |
| 競合多（成熟市場） | ニーズ段階 | Benefit→教育コンテンツ→Advantage（最も長い旅程） |

## Phase 2：コンテンツ企画

AB3Cで言語化された戦略を「実際に伝えるコンテンツ」に翻訳するフェーズ。  
**ここが最もクリエイティブで、最も品質を左右する。**

### 2-1. コンテンツ比較表（競合調査）

**目的**：「必要で、かつ他にないコンテンツ」を発見する。

伝えるべきコンテンツ = 全体 − **ユーザーがすでに知っていること**

→ 業界の当たり前を1から説明し直すと「どこにでもあるサイト」になる。  
→ 環境調査によって「差分」だけを伝える設計にする。

**作り方（帰納的積み上げ）：**

1. 自社の既存コンテンツをベースにカテゴリを設計（新規は仮カテゴリ）
2. 競合サイトを順に見て、自社にないコンテンツを行として追加
3. ○◎×でマーキング（◎=特に充実・独自性あり）
4. 必要に応じてカテゴリ自体を新設
5. 備考欄に定性評価を記入

**表の読み方：**

| 読み方 | 意味 | 設計への示唆 |
|---|---|---|
| 全サイト○ | 業界の当たり前 | 下層でよい（プル型） |
| ◎が自社だけ | Advantage候補 | 動線上に仕掛ける（プッシュ型） |
| 他社◎・自社× | 不足コンテンツ | 補完を検討 |

### 2-2. プッシュ/プル分類

コンテンツを「届け方」で2分類する。

**プル型（ユーザーが探しに来る）**
- ユーザーは存在を想定している——ご利用ガイド・FAQ・お問い合わせ等
- 深い階層に置いてよい
- ナビゲーションとカテゴライズで発見できれば十分

**プッシュ型（仕掛けなければ届かない）**
- ユーザーは存在を知らない——**Advantage系コンテンツがここに入る**
- ユーザーの動線上に「気配」として配置し、クリックを誘う
- カテゴリの奥に入れても誰も辿り着かない

> **原則：Advantage系コンテンツは必ずプッシュ型で設計する**

### 2-3. ユーザーモデル別コンテンツマッピング

3パターンのユーザーモデルに対して、各コンテンツを割り当てる。

| コンテンツ | パターン1（主軸） | パターン2 | パターン3 |
|---|---|---|---|
| 戦略メッセージ（H1） | ◎ | ○ | △ |
| 共通の信頼コンテンツ | ◎ | ◎ | ◎ |
| パターン1固有のAdvantage | ◎ | − | − |
| パターン2固有のAdvantage | − | ◎ | − |
| パターン3固有のAdvantage | − | − | ◎ |

→ **共通コンテンツ**と**ユーザーモデル固有コンテンツ**が明確になる。

**重要：Advantageは意思決定の「前」に届ける**

近江牛.com実例：
- ギフト用のAdvantage = 優れたパッケージング
- 通常設計では「商品選択→カート→最後にギフト包装オプション」（遅い）
- 近江牛.comの設計：グローバルナビで「自宅用/ギフト用」を最上位で分岐
  → ギフト購入者はサイト入口から自分向けのAdvanageが見える

## Phase 3：サイト構造設計

### パターン1（最重要ユーザー）でサイト骨格を設計

- H1・ヒーローはパターン1のBenefitとAdvantageで設計
- 主動線はパターン1のシナリオに沿って設計
- パターン1の邪魔をしないようにパターン2・3を組み込む

### パターン2・3の組み込み方

| ユーザーモデルの距離 | 組み込み方 |
|---|---|
| パターン2（近い） | 共通コンテンツはそのまま。固有コンテンツを専用セクションとして追加 |
| パターン3（遠い） | グローバルナビに分岐ボタン。別LP/サブサイト的に独立 |
| 並列型（同商品・違う目的） | タブ/バリアント形式 |

### AB3C5要素とサイトセクションの対応

| AB3C要素 | 訪問者の問い | サイトセクション |
|---|---|---|
| 戦略メッセージ | これは自分に関係あるか？ | ヒーロー・H1 |
| Customer（課題） | 自分の悩みを分かってもらえるか？ | 課題共感セクション |
| Benefit | 何が解決されるのか？ | 解決策・価値説明 |
| Advantage | なぜここなのか？ | 差別化・根拠セクション |
| Company（強み） | 信頼できるか？ | 実績・人物・第三者評価 |

## Phase 4：コンテンツ詳細化と担当決定

各コンテンツを種別・工数・担当で詳細化する。  
**ここまでできて初めて、アクションにつながる。**

### コンテンツ種別と判断基準

| 種別 | 判断基準 |
|---|---|
| テキスト | AIが最も得意。ほぼすべての説明・コピー |
| 写真 | 「このビジュアルがなかったら伝わらないメッセージがあるか？」→ Noなら不要 |
| イラスト/SVG | ブランドの素材言語がある場合（和紙・墨・木等） |
| 動画 | Advantageが動きでしか伝わらない場合のみ |
| AI生成画像 | 上記の写真基準を通過した場合のみ。「生成できる」≠「使うべき」 |

### 担当者アサイン

| 担当 | 対象コンテンツ |
|---|---|
| **AI自動生成** | テキストコピー・FAQ・構造説明・Benefit説明等 |
| **AIがインタビューから生成** | 経営者の想い・Advantage根拠・ビジョン等（人の言葉を起点に整形） |
| **クライアントが提供** | 人物写真・現場写真・実績データ・ロゴ・証明書等 |

### 工数・費用の見積り（生成コスト）

| 種別 | コスト目安 |
|---|---|
| テキスト生成 | API費用のみ（数十円〜数百円/ページ） |
| AI画像生成 | ¥3〜8/枚（Flux/DALL-E 3） |
| クライアント提供素材 | 撮影費用・準備時間（クライアント側負担） |

## Phase 5：生成・実装

### デザインシステム確認（フルページ生成の前に必須）

全部作ってから直す方式は「使う気をなくす」（権さん指摘）。  
先に決めることで生成品質が安定する。

```
1. パレット案（2〜3案）：色スウォッチ + 実際のテキストサンプル
2. フォント案（2案）：同じテキストで比較
3. ヒーロートーン（明/暗/中間）：ヒーロー部分のプレビュー
   ↓
ユーザーが選択（例：パレットA・Noto Serif JP・ヒーロー暗）
   ↓
フルページ生成（確定した設計で一気に）
```

### 技術スタック（確定）

```
Astro（HTMLに近く・LLM生成精度が高い・ビルド高速）
+ Tailwind CSS（CDN or Astroに統合）
+ shadcn/ui デザイントークン語彙（Reactコンポーネントは不要）
+ 画像生成API：Flux（Replicate）/ DALL-E 3（状況で選択）
```

## 2パス設計：簡易版と詳細版

### 簡易版（スタンダードパス）

```
AB3C確定
  ↓
デザインシステム確認
  ↓
自動生成（Phase 2〜4をAIが簡略実行）
  ↓
微調整・公開
```

- Phase 1→5を最短で通過
- コンテンツ比較表などはAIが内部で処理（ユーザーには見えない）
- **対象：ひとりでやる経営者**

### 詳細版（伴走パス）

```
AB3C確定
  ↓
コンテンツ比較表作成（競合調査）
  ↓
プッシュ/プル分類
  ↓
ユーザーモデル別コンテンツマッピング
  ↓
コンテンツ詳細化・担当決定・素材収集
  ↓
デザインシステム確認
  ↓
フルページ生成→反復調整→公開
```

- 全フェーズを人（コンサルタント・制作会社）と一緒に行う
- コンテンツの解像度が高く、戦略の純度が保たれる
- **対象：伴走支援者がいる場合・重要なサイトの場合**

## ⑪テストで確認されたこと（2026-06-07）

### Claude Code 単体で実用レベルのサイトが生成できる（確認済み）

gonweb.co.jp・nazuke.ai の2サイトでテスト実施。以下の組み合わせで「売り物になるデザイン」に到達した。

**品質を決める2つのレバー：**

**① デザインシステム（コンポーネントライブラリ）の活用**  
shadcn/ui のようなデザイントークン・コンポーネント語彙を使うことで、素のCSSでは出せない整合性と情緒が生まれる。使うのはReactコンポーネントではなく**視覚言語（トークン・構造）のみ**。ボタン・カード・バッジ・セパレーター等の基本パーツが揃っているだけで、全体のクオリティが底上げされる。

**② AIが生成できる装飾モチーフの活用**  
デザインシステムの外側に、AIが独自に生み出せる装飾がある。nazuke.ai での水墨の染みパターン（`blur` + `opacity` の組み合わせ）がその実例。ブランドの素材言語（和紙・墨・木・革等）をCSSで表現することで、外部画像なしに世界観を作れる。

```
デザインシステム（shadcn語彙）= 骨格・整合性
AIが生成する装飾モチーフ    = 世界観・情緒
この2つを組み合わせることで、テンプレート感のない固有のサイトになる
```

### 「一回で生成する」原則（重要）

AIに制約（「この写真をここに入れて」「この幅で」等）を与えると、AIが全体バランスを取れなくなる。  
**→ 極力コンテンツを先に収集し、一回で生成させることが品質の最大条件。**

| ケース | 難易度 | 理由 |
|---|---|---|
| **既存サイトのリニューアル** | 比較的容易 | コンテンツ（写真・文言・実績）が既に存在する |
| **新規サイト** | やや難 | 写真など素材の用意をクライアントに依頼する必要がある |

**新規サイトの素材要求フロー：**
1. Phase 2（コンテンツ企画）で必要な素材を特定
2. 担当者アサイン（クライアント提供 / AI生成 / インタビューから生成）を確定
3. 全素材が揃った段階でフルページ生成を実行（途中で生成しない）

## 付記：ウェブサイト生成機能の設計原則

1. **戦略はUIレベルで固定する**——デザインの好き嫌いが戦略変更にすり替わらないよう制御
2. **AB3Cラベルはサイトに出さない**——B/Aはフレームワーク内部の言語。ユーザー向けには自然な見出しで
3. **テンプレートは使わない**——AB3Cの独自性がそのままサイトに反映される設計
4. **素材の乱造をしない**——「このビジュアルがなかったら伝わらないメッセージがあるか？」で判断
5. **デザインシステムを先に確認する**——全部作ってから直す方式はユーザーの使う気を失わせる
