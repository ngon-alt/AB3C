## 1. 目的とコンセプト

楽天市場の店長が、自分の商品LPのURLを入力すると、次の三段が一本の物語として返ってくる。

| 段 | 問い | 中身 |
|---|---|---|
| 第一段 | **売れている店は、なぜ売れているのか** | 競合上位商品のレビューから「選ばれる理由」を採掘して見せる |
| 第二段 | **自分は、なぜ売れないのか** | 自社LPの訴求と顧客評価のズレ、競合との強み弱み比較、LPの語り方の欠落 |
| 第三段 | **では、どうすればいいのか** | 訴求ポイントの再設定 → 戦略メッセージ確定 → コンテンツ増強の処方箋 |

分析部品を並べるレポートではなく、「なぜ → なぜ → ならば」の物語として書く。これは戦略指南AI本体で起きた「経営・戦略の知識がない人には中身が理解できない」問題への直接の答えである。店長は自分の商売の物語として読める。

### 設計の根本前提（権さん確認済み・2026-06-12）

- **ニーズ/ウォンツは商品の属性ではなく、購入者の段階である。** 同じミネラルウォーターでも「ミネラルウォーター」と検索する人（ニーズ段階）と「ボルヴィック」と検索する人（ウォンツ段階）がいる。
- ブランド品であればウォンツ段階の購入者が多いだろうと言えるが、楽天で売られる商品の大半は**ニーズ段階の購入者が多い**。
- **「ニーズ型前提で統一する」の真意は、商品の分類ではなく「説得の設計」の統一である**（権さん補足・2026-06-12）:
  - ウォンツ型（型番・指名買い）の商品は、商品本体は競合とスペックが同一のため、価格・納期だけで判断される。
  - しかし**仕入れ品の売り手でも、サービス（購入前・購入中・購入後）と「外側のパッケージ」は自分たちの提供物**であり、ここで差別化が作れる。例: 大型家具の購入時、有害物質付着で通常廃棄できない旧家具を回収・廃棄してくれる店——サービスによるAdvantageの実例。外側のパッケージの例: おまけの同梱、配送用段ボールの工夫、剥がしやすいガムテープの貼り方。
  - **用語注意（権さん指摘・2026-06-12）**: ここで言う「外側のパッケージ」は、価値サークルの「パッケージング」軸とは**別物**。価値サークルのパッケージングは**商品そのもののベースパッケージ**（商品の一部）を指す。仕入れ品の売り手が工夫できるのは、その外側にある包装・同梱・配送まわり＝**サービスに近い領域**である。
  - そしてサービスの価値は黙っていては伝わらない。**コンテンツで説明して選んでもらう＝ニーズ型と同じ「コンテンツによる説得」の設計が必要になる**。同じ商品を売る店の中から「ここのサービスがいい」と選んでもらう構造。
  - だから全商品を同じエンジン（ニーズ型の説得設計）で分析する。入口での機械的なモデル判定・処理分岐は**置かない**。ウォンツ型商品は商品本体のスペック差が抽出されないため、分析結果は自然にサービス・パッケージング軸の強みに収束する（構想ドキュメント第3章の合意どおり）。
- ただし**説明の言葉として「ウォンツ型」を使うのは推奨**。型番・指名買い色の強い商品には「この商品はウォンツ型（指名買い）のため、商品本体は価格・納期で判断されます。差別化はサービスで作り、その価値をコンテンツで伝えて選んでもらいましょう」という言い方をする。分類用語としてではなく、店長への説明をわかりやすくする道具として使う。
- ニーズ段階前提の理論的裏付け＝「モール出店は、他店の同種商品とおのずと比較され、相対的なウォンツが形成される装置」（『4モデル式』p.88）。楽天の商品LPは構造的に**比較シーンの中で読まれるページ**であり、分析は単独ページの良し悪しではなく競合との相対評価で行う。

## 3. 入力仕様

### 3-1. 入力の流れ

```
① 自社の商品LP URL を入力
② カテゴリーを指定（前回値が自動で入る。上書き可）
③ 競合候補リストが提示される → チェックで選ぶ（外す・足す・自由URL追加可）
④ 分析実行
```

### 3-2. カテゴリー登録（大・中・小の3階層）

- **大カテゴリー・中カテゴリー・小カテゴリーの3階層を選択させる**（権さん指定・2026-06-12）。
- 店長本人に指定させる。操作の手間ではなく「自分の商品はどの戦場にいるのか」を意識させる**教育効果**を狙う（構想ドキュメント第7章と同思想）。3階層を自分で選ぶこと自体が「大カテゴリーでは激戦のレッドオーシャン、小カテゴリーなら競合数件のニッチ」という戦場の構造を見る体験になる。
- **店舗単位でデフォルト保持**: 一度入力したカテゴリー（3階層とも）はサイト（店舗）に紐づけて保存し、次回の商品URL入力時に自動で埋める。消したいときは上書きできる。小規模店はほぼ全商品が同一カテゴリーという実態に合わせる。
- カテゴリーは競合母集団の特定（3-3）と、価値軸の第2層（4-2）の両方に使う。競合候補の提示はまず小カテゴリーのランキングを基本とし、必要に応じて中・大に広げる（粒度の使い分けの詳細は実装時に調整）。

### 3-3. 競合候補の自動提示 → 店長が主体的に選ぶ

- 指定カテゴリーのランキングから競合候補を自動で並べて見せ、店長がチェックボックスで競合を確定する。
- **候補には順位・価格・レビュー件数・平均星数（★）を添える。**「1位だが価格が1/3（比較対象として不適かも）」が一目で分かるようにする。ランキング上位＝適切な競合とは限らない（激安上位は学べない）ため、比較に値するかの判断は店長に残す。
- 平均星数は選択基準というより**学習のため**（権さん指定・2026-06-12）。件数×星数の組み合わせから市場の構造が見える（例: 件数多×★3.8＝売れているが不満も多い／件数少×★4.8＝小さいが熱烈に支持される）。レビュー分析に入る前の「この競合はどんな選ばれ方をしているか」の予告編として機能する。公式APIがreviewCount・reviewAverageを両方返すため追加取得コストはない。
- 自由URL追加を許す。「本当はこれと比べたい」という指名比較に対応する。
- この「候補から選ぶ」行為自体が、自分の戦場を眺める教育体験になる。
- 取得経路はランキングの公式API（楽天ウェブサービス）を本命とする（第7章）。
- 競合数の目安: 自社1 + 競合3〜5商品（レビュー取得・トークンコストとのバランス。上限は実装時に調整）。

## 4. 分析パイプライン

```
[自社LP取得] ──┐
[競合LP取得] ──┼→ [レビュー層化抽出] → [価値軸の仕分け] → [ズレ可視化] → [AB3C・戦略メッセージ] → [三段の物語レポート]
[レビュー取得] ─┘
```

### 4-1. レビューの層化抽出（分析の心臓）

- 各商品につき、件数を均等に取るのではなく**評価で層化**する: 高評価（★4〜5）から数十件＋低評価（★1〜2）から数十件。「選ばれる理由」と「離れる理由」のコントラストが少ない件数で鮮明に出る（構想ドキュメント第8章）。
- 商品ごとに一度要約してから束ねる**二段構え**（コンテキスト上限対策）。
- トークンコストは1分析あたり数百円のオーダーで制約にならない。件数を絞る理由はコストではなく精度と鮮度。

### 4-2. 価値軸の抽出と仕分け（価値サークル6軸）

レビューの一文一文を権さんの**価値サークル**の6軸に仕分けて、自社と競合の強み弱みを構造化する。

- 商品力の3軸: **機能（商品本体）／デザイン／パッケージング**
- サービスの3軸: **購入前サービス／購入中サービス／購入後サービス**

**仕分け時の用語注意**: 「パッケージング」軸は**商品そのもののベースパッケージ**（商品の一部。例: ギフト用化粧箱）を指す。配送用段ボール・梱包の丁寧さ・おまけ同梱・開封しやすさ等の「外側のパッケージ」への言及は、パッケージング軸ではなく**サービス軸（主に購入中サービス）**に仕分ける。仕入れ品の売り手が差別化できるのは後者（第1章の用語注意参照）。

価値軸（比較の横軸）は3層で構成する（構想ドキュメント第4章）:
1. 全通販共通項目（価格・納期）— 事前固定
2. カテゴリー固有項目（食べ物なら味、衣類ならサイズ感）— v1ではAIがカテゴリーから推定
3. その商品に固有の価値軸 — レビューから抽出

### 4-3. 訴求と評価のズレ可視化（v1の核）

- 自社LPが訴求していること（LPテキストから抽出）と、顧客がレビューで実際に評価していること（4-1, 4-2の結果）を突き合わせる。
- 二段階で見る:
  1. 訴求と評価がずれていれば「訴求の力点が間違っていたのではないか」という気づき
  2. その顧客評価が競合と比べて本当に強みか確認 — 競合もみな「おいしい」なら強みではない。**自社だけ違うところに顧客が強みを認識しているなら、それは本物の差別的優位であり、伝えるだけで価値になる**
- レビューで繰り返される特徴語＝Advantage候補、という理論的裏付け: 購買率の高いキーワードは「商品の差別的特徴を表すキーワード」（『SEM戦略』p.81-82 の「基礎化粧品→ナノ化粧品」の論理の裏返し）。

### 4-4. LPの語り方診断（観点リスト v2・2026-06-12 議論反映）

ニーズ段階の購入者は「サイト訪問後に購買意欲を高められるか」が勝負であり、LPコンテンツの質が売上に直結する（『4モデル式』p.65）。自社LPの「語り方」を以下の**7要素**（優先順位つき）で診断する。

この7要素の並びは**AB3Cの物語順**と一致する: ①＝顧客（C）、②＝ベネフィット（B）、③＝アドバンテージ（A）、④＝Aの根拠、⑤＝Bの情緒的翻訳、⑥⑦＝行動への橋。LP診断＝「AB3Cの物語として語れているか」の診断である。

| # | 観点 | 診断する内容 |
|---|---|---|
| ① | **ターゲットの悩み** | 入口で「こんな悩みありませんか」と課題から入っているか。悩みへの共感（楽天らしい入り方でもある） |
| ② | **ベネフィット** | 開いた瞬間（ファーストビュー）に「何の商品で、なぜ買うべきか」が分かるか。楽天ユーザーは3秒で離脱する |
| ③ | **他社との違い** | 差別的優位点が語られているか。**競合比較の構図が分かる比較表**があるか |
| ④ | **証拠（信頼コンテンツ）** | 下記の4種で構成: （a）**レビューの量と質**＝「多くの人が選んでいる」安心。最大の信頼コンテンツ（5件以上で購入確率270%増等の定量裏付けあり）（b）**ガイドコンテンツ**＝売り手が専門家として「これはこういう意味で良い」と意味づけ・おすすめする説明（ヨドバシの店員体験。米国EC統計で売上貢献1位）（c）**第三者の証明**＝認証・検査・受賞・専門家推薦・メディア掲載（メディアは「多くの人が評価している」の意味づけで語る。「載りました」だけでは弱い）（d）**開発ストーリー**＝ナラティブで商品への信頼を伝える。レビューが少ない店は（b）（c）（d）で客観的信頼を補う |
| ⑤ | **使用シーン** | 論理だけでなく**情緒的に**伝えているか。「そのシーンなら、こう感じるな」を頭に想起させる |
| ⑥ | **不安解消** | FAQ（不安払拭）・保証・**基本情報**（スペック・配送納期・送料）がページ内で分かるか。ページ遷移しないと分からない情報は不安になる（追加費用の不明瞭さは離脱の最大要因の一つ） |
| ⑦ | **今買う理由** | クロージング。今このページで買う理由が提示されているか |

横断チェック: スマホ表示（楽天流入の7〜8割はスマホ）。

**実装上の扱い**: 観点リストはプロンプト内の差し替え可能な定義として実装し、運用しながら文言を磨ける構造にする（コード構造はリストの中身に依存させない）。

### 4-5. AB3C・戦略メッセージ（出口）

- ズレ分析・強み弱みを踏まえ、選んだ軸をAdvantageに据えたAB3C（Benefit／Advantage／3C）と戦略メッセージを生成する。
- 強み弱みは**両方提示**する。強い軸は訴求の中心に据えるよう背中を押し、弱い軸は「補うか、新しい強みを作るか」と問いを返す（Advantage不成立時は商品・サービス開発を提案する、というAB3C原則のとおり）。
- 「商品を買う理由」と「この店で買う理由」を分けて言語化する（『4モデル式』p.16-18, 186）。楽天の同一商品多店舗環境では後者（サブベネフィット: 同梱・サポート・ギフト対応等）が効く。

## 5. 出力仕様（レポート構成）

三段の物語をそのまま章立てにする。

### 第一段「売れている店は、なぜ売れているのか」
- 選んだ競合ごとに、レビューから採掘した「選ばれる理由」を提示（価値サークル6軸への仕分け結果つき）
- 競合間で共通する強み（＝このカテゴリーの土俵）と、各競合に固有の強み（＝各社の山）を分けて見せる

### 第二段「自分は、なぜ売れないのか」
- 自社の訴求 vs 顧客評価のズレ（4-3）
- 競合との6軸比較表（強み弱み両出し）
- LPの語り方診断（4-4・観点リスト確定後）

### 第三段「では、どうすればいいのか」——二段階の処方箋（権さん指定・2026-06-12）

処方箋は**「今すぐやるべきこと（対処療法）」と「本質的な対策（価値そのものを高める）」の二段階**で評価・表示する。店長が「今週やること」と「今期考えること」を混同しないようにする。

**第一段階: 今すぐやるべきこと（伝え方の改善）**
- 冒頭に「あなたが重視すべき、お客様の比較ポイント上位3つ」を提示（例: 味・パッケージ・価格）。お客様が比較している価値軸を列挙し、優先順位をつけて見せる。1〜5点の客観的な点数比較（戦略キャンバス）まではv1ではやらないが、**優先順位（順序）はレビューの言及頻度・熱量から大づかみの分析でも当てられる**
- その上位軸に合わせて、**重視するポイントを変え、LPの訴求・コンテンツを変えていく**: 訴求ポイントの再設定（「訴求のバランスをユーザーの重視点に偏らせる」『4モデル式』p.80）、コンテンツ増強の優先順位、LP語り方診断（7要素）の改善指示
- 戦略メッセージ（AB3C出口）の再確定

**第二段階: さらに本質的には（価値そのものを高める）**
- **価値サークルの6軸で「価値そのものを高める」抜本的対策を提案する**。例:「購入前サービスで、こういうものがあるといい」——伝え方ではなく提供価値の開発。AB3Cの「Advantageが立たなければ商品・サービス開発を提案する」原則の実装
- 「どの軸で戦うか」のシナリオ提示（複数の世界線。v1では簡易版＝主要2案程度）
- 全社が低い／誰も載せていない軸があれば**新しい軸の提案**（ブルーオーシャン。近江牛.comのパッケージ事例の構造的再現)

v2で戦略キャンバスを図示する際は、第一段階の上位軸がそのまま横軸の主役になる。

### 表示・言語ルール
- 戦略指南AI本体と同一: AB3Cカラー（C=黒 #1a1a14／B=赤 #FF0000／A=青 #1a6fd4）、本文18px以上、ok/warn/ng等の英語ステータス語をユーザーに見せない。
- 評価語は「整っている」「まだ弱い」「改善が必要」等の自然な日本語。

## 6. 採用しない・参考にしないと決めたこと

| 項目 | 理由 |
|---|---|
| **タイトル（商品名）へのキーワード有無による競合の本気度判定** | 『SEM戦略』のサープス分析由来の手法だが、**楽天内SEOとGoogle SEOは評価要素が大きく異なる**ため、タイトルのキーワードだけでは判断できない（権さん判断・2026-06-12）。本質的な競合を探すという視点のみ受け継ぎ、手法は移植しない |
| ニーズ/ウォンツの機械的モデル判定・処理分岐 | 第1章のとおり。全商品を「ニーズ型の説得設計」で統一分析（ウォンツ型は結果が自然にサービス軸へ収束）。「ウォンツ型」という言葉自体は店長への説明用語として使う |
| ランキング順位 vs 楽天サーチ検索順位の突合 | 楽天内SEOの評価要素に依存する診断のため、v1では扱わない |
| 検索キーワードベースの競合発見 | 楽天はカテゴリー回遊が主流（構想ドキュメント第2章の結論どおり、競合発見はカテゴリーランキング軸） |

## 7. 技術仕様

### 7-1. データ取得（検証結果: 2026-06-12 スパイク）

| 対象 | 経路 | 検証結果 |
|---|---|---|
| レビュー本文 | `review.rakuten.co.jp/item/1/{shopId}_{itemId}/1.1/` を直接取得 | **成功**。ブラウザ相当のUAヘッダー付きならHTTP 200。ページ内 `window.__INITIAL_STATE__` のJSONにレビュー本文（body）・評価が構造化されて入っており、HTML解析不要 |
| 商品LP | `item.rakuten.co.jp` を直接取得 | 未検証（実装初日に確認） |
| カテゴリーランキング | 楽天ウェブサービス（公式API） | ページ直接取得は403のためAPI経路が本命。アプリID取得が必要（無料）。APIのitemCodeからレビューページの数値ID（shopId_itemId）への橋渡しは実装初日に検証 |

**リスクと留意点:**
- 検証はローカル回線から実施。**VercelのデータセンターIPからは403になる可能性が残る**。実装初日に本番経路で確認し、不可なら軽量プロキシ（中継）を挟む。
- 標準的なfetch（プログラム的UA）は403で弾かれるため、ブラウザ相当のUAヘッダーが必須。
- ページ販売単位のエンコーディングに注意（旧ページはEUC-JP混在の可能性）。
