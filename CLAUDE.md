## プロジェクト概要
「戦略指南 AI」は、権成俊氏が20年以上のコンサルティング実績から開発したAB3Cフレームワーク（Advantage・Benefit・Customer・Competitor・Company）をSaaS化した戦略策定ツール。非エンジニアながらClaudeと協働開発しており、日本の中小企業に戦略コンサルティング品質の分析を届けることをミッションとしている。

### 本質的差別化（重要）
他のフレームワーク系ツール（3C・4P・5フォース等のAI自動分析）と決定的に違うのは、**「分析→新事業プラン作成→アクション実行」までの一気通貫フロー**を提供すること。「分析しっぱなし」では満足度が出ない。権さん自身が成果を出している実フローをそのまま実装している。
**自己分析（経営者の価値観・原体験の言語化）はサービス体験の中核**。本人の気づき＝アイデンティティ確立＝自信＝戦略を堂々と打ち出す力、という連鎖を支援する。

## AB3Cフレームワークの理解（重要）

### 基本構造
AB3C分析は「選ばれる理由」を明らかにする事業戦略フレームワーク：
- **B (Benefit)**: お客様が求める価値（ニーズ→ウォンツ）
- **A (Advantage)**: 差別的優位点・好ましい違い（競合より選ばれる理由）
- **3C**: Customer（顧客）・Competitor（競合）・Company（自社）
- **戦略メッセージ**: Benefit + Advantage の統合表現
- **チェックポイント**: AB3Cの整合性を5項目で評価（ok/warn/ng）

### AB3Cフレームワーク原則（コース全7回学習済み）
1. **ビジョンから始めよ**: 大きな石から入れる、ビジョン→戦略→戦術の順
2. **戦略とは**: 問題の本質を見極める
3. **AB3Cで戦略を表現**: 全体と部分を一緒に表現、BenefitとAdvantageをデザインに直結、一気通貫
4. **調査分析**: レビュー・検索KW・VRIO分析でAB3C全要素を調査
5. **商品・サービス企画**: AB3C不成立時は商品開発を提案、価値サークル、業界常識を壊す
6. **コンテンツ企画**: AB3C→ユーザーシナリオ→CJM→コンテンツの順
7. **何者になるか**: 実行できない本質はビジョン欠如、パッション→ポジション→ミッション

### 分析時の重要原則
- ビジョンから問う
- 問題の本質を掘る（表面的課題の奥を問う）
- BenefitとAdvantageを明確に分離
- **一気通貫**（Web・商品・営業・採用すべてに同じ戦略軸）
- 競合視点重視（「好ましい違い」か問う）
- Advantage不成立時は商品・サービス開発を提案
- 経営者の価値観を3層で掘る（外の違い→理由→価値観）
- 業界の常識に疑問を持つ
- **Advantage の根拠を問いかけ型で促す** — 第三者証明は必須ではないが、本人が確信を持って語れる根拠（独自経験・先駆者性・実績等）があるかを確認・言語化を促す。「最高」「一番」のような主観的すぎる主張は本人なりの根拠の整理を促す。スタンスは断定ではなくコーチング型（「〜の根拠を整理しておくと信頼性が高まります」）

### 分析結果JSONスキーマ
```
benefit: { core, needs[], wants[] }
advantage: { what, why_good, why_hard_to_copy }
three_c: {
  customer: { target, profile[], stage, cutoff, market: { sam, som, growth, basis } },
  competitor: { direct[], indirect[] },
  company: { strength[], structure, passion }
}
strategy_message: { message, benefit_part, advantage_part }
checkpoints: [{ label, status, comment }]
```

### AI生成テキストの言語ポリシー（厳守）
- **内部の enum 値（`ok` / `warn` / `ng` 等）はユーザーに直接見せない**。UI 側で「問題なし」「注意」「警告」等の日本語表記に変換する
- **AI が生成する comment / 本文に英語の状態ラベル（`warn`, `[warn]`, `ok`, `ng` 等）を含めない**。評価の度合いは「問題なし」「要注意」「改善が必要」「整っている」「まだ弱い」等の自然な日本語で表現する
- **一般に馴染みの薄い英単語・略語（VRIO, KPI, CVR, CPM 等）は避ける**。使う必要がある場合は日本語で噛み砕いた言い換えを併記（例: 「VRIO分析」→「真似しづらさの分析」）
- 既に広く使われているカタカナ語（ウェブサイト、ターゲット、コンテンツ、ベネフィット、アドバンテージ、ブランド、メッセージ等）は使用可
- プロンプト変更時もこのポリシーを保持すること

### フォント定義（globals.css に CSS 変数として定義済み）
```

--font-heading: 'Noto Serif JP', serif（明朝） ← 見出しのみ

```
- body のデフォルトは `--font-body`（ゴシック）

- **新規UIで `fontFamily: "'Noto Serif JP', serif"` を直接指定する場合は「これは見出しか」を必ず確認**
- ページ全体の wrapper（`<main>` `<div>`）に `fontFamily: serif` を入れない（サブツリー全体が明朝になる）

## 技術的な重要教訓
- `.env.local`等の秘密情報をコミットした場合: `git reset --soft HEAD~1` → ファイル除去 → 再コミット（force pushは不可）
- `str_replace`ツールは大きなファイルで文脈が曖昧だと失敗する。十分にユニークな周辺コードを検索文字列に含める
- `page.js`にインライン記述されたFooterコードはFooter.jsとは別物（ページ固有のインラインコードを先に確認する習慣）
- `page.js` は巨大ファイル（1400行超）。読み込み時はoffset/limitを使用
